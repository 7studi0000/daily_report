### 1. わかったこと (核心的な理解)

🔸 **Sourcesパネル＝読み込み資産の司令塔**

* **ファイルナビゲーターの見方:** ページが実際に読み込んだ HTML / CSS / JS / 画像がすべて一覧される。ここに無い＝リンク切れや読み込み失敗、という切り分けができることを理解した。
* **コードエディター＋デバッガーの三位一体:** Sourcesは「ファイル確認」「コード閲覧」「JavaScriptデバッガー」の3領域で構成され、フロントエンド実装とデバッグが同じ画面で完結する設計だと体感した。

🔸 **JavaScriptデバッガーによる「処理の可視化」**

* **Step over と Step into の使い分け:** Step over は関数を「結果が返ってきたもの」として1行ずつ進み、Step into は関数の中に潜って引数・戻り値まで追う。`console.log` では見えにくい実行フローそのものを観察できる点が決定的に強い。
* **Scope と Watch:** ブレーク中に Local / Global で変数の現在値を確認し、Watch で特定変数を常時監視できる。型のズレ（number のはずが string など）にも気づける。
* **ブレークポイントの種類:** 行番号クリックでの停止に加え、Mouse Click などのイベントリスナーブレークポイントで「きっかけ」から処理を止められる。Elements側の DOM ブレークポイントとも連携して使えると分かった。
* **処理のムダに気づける:** if 連鎖で条件が当たったあと `return` しないと、以降の if も評価され続ける——デバッガーで一歩ずつ進めると、頭の中の想定と実際の流れのズレが一発で見える。

### 2. 次やること (自力で発見する課題)

* **自前コードでのステップ実行:** おみくじサンプルではなく、自分の小さな JS（イベント＋条件分岐＋ループ）にブレークポイントを打ち、Step into / Watch で変数の変化を追う練習をする。
* **Applicationパネルの予習・実践:** 動画の次回予告どおり、LocalStorage / SessionStorage / Cookie など Application パネルを触り、フロントの状態管理デバッグにつなげる。
* **Networkパネルとの接続:** Sourcesで「どのJSが動いたか」と Networkで「どのAPIが飛んだか」をセットで見る習慣をつけ、外部API連携のデバッグフローを完成させる。

### 3. 感想 (内省)

前回の Elements / Console が「理論を透視する眼鏡」だったのに対し、今回の Sources デバッガーは「処理の時間軸を止めて観察する顕微鏡」を手に入れた感覚です。
