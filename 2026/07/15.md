### 1. わかったこと (核心的な理解)

🔸 **ブラウザ内ストレージの使い分け**

* **Cookie / LocalStorage / SessionStorage:** Cookie は有効期限あり・容量小。LocalStorage は有効期限なしで比較的大容量（おおむね数〜10MB級）かつ高速なキーバリュー保存。SessionStorage はタブ／ウィンドウが開いている間だけの一時保存で、閉じると消える。用途（永続設定 vs 一時状態）で選ぶ設計だと理解した。
* **オリジン単位の隔離:** 保存データは「プロトコル + ドメイン + ポート」のオリジンごとに分離される。別サイトの LocalStorage は読めない。ログイン状態や設定がサイト単位で残る仕組みの土台だと体感した。
* **IndexedDB と廃止機能:** IndexedDB はより本格的なクライアント側DB。WebSQL や Application Cache は標準から外れ、新規では使わない時代だと整理できた。

🔸 **Applicationパネル＝PWAとキャッシュの観測窓**

* **manifest.json:** サイト名、アイコン、起動URL、テーマカラー、`display: standalone` などを定義し、「インストール可能なWebアプリ」にする設定ファイル。DevTools 上で中身を確認できる。
* **Service Worker:** 裏でキャッシュ対象ファイルを制御する。ステータスが緑＝稼働、赤＝エラーでキャッシュも効いていない、という見方が分かった。
* **Cache Storage と Clear site data:** キャッシュされたファイル実体と更新日時を確認でき、「リロードしても日時が変わらない＝キャッシュ側を読んでいる」と判断できる。Storage からサイトデータを一括消去してデバッグ用にクリーンに戻せる。

### 2. 次やること (自力で発見する課題)

* **LocalStorage の自作ハンズオン:** セレクトやトグルの値を `localStorage` に保存し、リロード後も状態が残る小さなUIを自分で書いてみる。
* **実サイトの Application 観察:** よく使うサイトで Cookie / LocalStorage / IndexedDB / Service Worker の中身を眺め、「何を永続化しているか」を観察する習慣をつける（個人情報は他人に見せない）。
* **次回 Lighthouse の予習:** 動画予告どおり、パフォーマンス／SEO などの採点機能に触れ、Application で見たキャッシュ戦略が点数にどう効くかの接続を意識する。

Elements が「見た目」、Sources が「処理の時間軸」なら、Application は「ブラウザに残る状態」を透視するパネルでした。前回「次やること」に挙げていた LocalStorage / Cookie の実践が、そのまま本編で腹落ちした感覚があります。
